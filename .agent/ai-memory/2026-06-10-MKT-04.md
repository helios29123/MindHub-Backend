# MKT-04: Quản lý banner

## 1. Tóm tắt nhiệm vụ
- Xây dựng các API CRUD cho banner dưới tiền tố endpoint `/api/admin/campaigns` và `/api/admin/campaigns/{id}` dành cho vai trò quản trị viên (`Admin`).
- Đáp ứng một phần: Cơ sở dữ liệu chỉ có bảng `banners`, không có bảng `campaigns`. Do đó toàn bộ các tác vụ CRUD (Create, Read, Update, Delete) sẽ được thực hiện trực tiếp trên bảng `banners` thông qua Model `Banner` có sẵn.
- Hỗ trợ soft delete khi xóa banner (bảng `banners` có cột `deleted_at` và Model `Banner` sử dụng `SoftDeletes`).

## 2. Quyết định kỹ thuật & Thiết kế
- **Xác thực & Phân quyền (Middleware)**:
  - Route được bảo vệ bởi middleware `['auth.session', 'role:admin']` đảm bảo chỉ Admin mới có quyền thực hiện CRUD.
- **Hợp nhất Controller Method**:
  - Toàn bộ các HTTP verbs (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`) được ánh xạ chung đến một phương thức duy nhất `banners()` trong `MarketingController` để xử lý phân luồng CRUD dựa trên HTTP request method.
- **Validation**:
  - Tạo mới request validator `BannerRequest` xác thực đầy đủ các thuộc tính của banner.
  - Ghi đè phương thức `failedValidation()` để trả về thông điệp lỗi cụ thể theo nghiệp vụ yêu cầu:
    - Nếu sai trạng thái: Trả về lỗi 422 kèm message `"Trạng thái banner không hợp lệ."`.
    - Nếu ngày kết thúc trước ngày bắt đầu (`end_at < start_at`): Trả về lỗi 422 kèm message `"Thời gian banner không hợp lệ."`.
- **Định dạng dữ liệu trả về (Resource)**:
  - Sử dụng `BannerResource` để format dữ liệu trả về đối với các tác vụ lấy danh sách (GET paginated) hoặc xem chi tiết (GET detail).
  - Đối với các tác vụ tạo mới (POST) và cập nhật (PUT/PATCH), API trả về dữ liệu kiểu JSON stringified dạng `{"banner_id": 1, "status": "active"}` lưu tại trường `data` khớp chính xác với đặc tả kỹ thuật.
